Design-Time Components

The design-time components consist of the User Interface Module and the Plugin Module. These components create the wizard. The runtime components use the data accepted by the wizard to connect to the target application and perform selected operations or request/response tasks.

You build the design-time components using the Adapter SDK APIs listed in the following table.

Wizard Step

User Interface Module Actions

Plugin Module Actions

1. Wizard Initialization

Wizard is initialized based on values defined in extension.xml or cloud-adapter.xml. Examples:

JDeveloper — extension.xml, param: implementation-class

ICS or SB — cloud-adapter.xml, param: UIProviderClass

None

2. Connection page

i) Create a new instance of the plugin.

ii) Get CloudConnection interface from the plugin.

iii) Request user input for connection properties (such as username, password, WSDL URL, and so on).

iv) Invoke the ping() function on the CloudConnection interface (from the plugin).

Note: You don’t have to write a completely new connection page for each adapter. You can use CloudConnectionPage from the SDK, and page controls can be derived from the cloud-adapter.xml file.

i) Instantiate the plugin with information passed in from AdapterPluginContext

ii) Return an instance of the CloudConnection interface.

iii) Facilitate invocation of the ping() function (using Connection interface).

3. Metadata browsing page

i) Get an instance of CloudMetadataBrowser from the plugin.

ii) Invoke the loadMetadata() function on the CloudMetadataBrowser instance.

iii) Display a dropdown of APIs from the CloudMetadataBrowser instance.

iv) Display a list of operations from the CloudMetadataBrowser instance based on the selected API.

v) Display a list of business objects from the CloudMetadataBrowser instance based on the selected operation.

vi) Facilitate the setting of header and processing options based on the selected operation from the CloudMetadataBrowser.

i) Return an instance of CloudMetadataBrowser.

ii) When the load() function is invoked, any custom metadata parsers should process the necessary artifacts. In the simplest case, metadata is parsed from source artifacts such as WSDL files. The plugin framework already has parsers for these artifacts.

iii) Store all objects (operations, objects, faults, and so on) in the CloudApplicationModel, which is wrapped by the CloudMetadataBrowser.

4. Query support (optional)

i) If the user selected a Query operation in #3, provide a UI for user to enter a query string, plus contextual help.

ii) Provide a test button for the wizard user to test a query.

ii) Provide a UI for the wizard user to enter values for bind variables.

i) Provide an instance of QueryBuilderHelper for contextual help.

ii) Provide support for parsing bind variables. A default implementation of QuerySupport, which can be extended, provides basic support for this feature.

iii) Provide support for testing queries.

5. Artifact generation

i) Create an instance of the Transformation Model, which contains all selected objects and changes ("transformations") that need to take place during generation.

ii) Create file system abstractions for where generated files need to be written.

iii) Get an instance of RuntimeMetadataGenerator from the plugin.

iv) Set the TransformationModel on the RuntimeMetadataGenerator instance and invoke the generate() method.

i) Return an instance of RuntimeMetadataGenerator.

ii) Generate artifacts when the generate() function of RuntimeMetadataGenerator is invoked. There are built-in artifact generators for the integration WSDL and cloud application configuration (CAC) file. Any additional files can also be generated here if necessary.
