pcmOpArItemDispute

Componentfm_ar
Visibilitypublic
Transactionrequired

Description

This operation makes dispute against an item's deferred or due amount. Following are the ways that an item will be disputed: The item dispute amount must be specified. The amount can be a positive or a negative number. A negative amount credits an item while a positive amount debits it. The amount will be converted to the account's primary currency for the dispute. The item due amount must not be zero before the dispute or becomes negative after the dispute including the taxes. If a back date is specified, it must be on or after the account and the general ledger (GL) posting date. The dispute process involves closing the bill with the item's bill-info, creating dispute item per usage amount and taxes, analysis usage (act_usage) for balance impacts, record events, and finally transfer the disputed amount and taxes. Caller can also explicitly specified an dispute item to avoid the operation from creating one. If item is open or closed, the operation will process the cycle tax (if any). If the item is pending, the operation will process the real-time tax (if any) when the PIN_AR_WITH_TAX flag is specified. Notes on tax treatment: The operation reads the tax flag from optional field PIN_FLD_FLAGS specified in the request. If the value of the flag is PIN_AR_WITH_TAX then tax is calculated on the amount specified in the request. If the value of the field is PIN_AR_WITHOUT_TAX then only the amount specified in the request is disputed. If the PIN_FLD_FLAGS optional field is not specified in the request then the system's default tax behavior (PIN_AR_WITH_TAX) is used. This setting is specified in the cm pin.conf as follows: - fm_ar tax_reversal_with_tax 1 #for without tax - fm_ar tax_reversal_with_tax 2 #for with tax Notes on the operation flags: When the flags include the pcmOpflgCalcOnly flag , the operation will skip creating a dispute item, transferring the dispute amount, and writing the modified item object fields to the database. But the fields that would have been used to create the event object are returned to the caller on the response. When the flags include the pcmTransOpenLockObj flag, a transaction will be opened to lock the item object. When the flags include the pcmOpflgReadResult flag, all the fields in the event object are returned on the response, not just the poid. Notes on the alternation to the operation request: The operation will add a PIN_FLD_BILLINFO_OBJ field and a PIN_FLD_BAL_GRP_OBJ field. The operation will alter the PIN_FLD_AMOUNT and PIN_FLD_CURRENCY fields and backup the values in the IN_FLD_AMOUNT_ORIG and PIN_FLD_RESOURCE_ID_ORIG fields The operation will set the PIN_FLD_FLAGS field if it is not provided with the tax_reversal_with_tax pin.conf configuration value. The default value is PIN_AR_WITH_TAX. One of the following operation operation description messages can be returned to the caller: Item is tax only item and can't dispute it. Dispute not allowed since backdate is before account effective_t. Dispute not allowed since backdate is before GL posting date. Item due is zero, cannot dispute item. Dispute amount is greater than item due, cannot dispute item. Dispute amount will be greater than item due after applying taxes, cannot dispute item. The PIN_ERRCLASS_SYSTEM_DETERMINATE error buffer returned (if any) can be: PIN_ERR_BAD_OPCODE

Result of the operation. Following are the values: Calculated as a bitwise OR operation of 1 with any of the following values: PIN_VALID_ITEM_STATUS_PENDING 0x02 PIN_VALID_ITEM_STATUS_CLOSED 0x04 PIN_VALID_ITEM_NO_TOTAL 0x08 PIN_VALID_ITEM_NOT_IN_DISPUTE 0x10 Considering that a given item can be either pending or closed, we will arrive at the following values: 3 - Item is pending. 5 - Item is closed. 9 - Item has no total. 11 - Item is pending with no total. 13 - Item is closed with no total. Given the item is open the value will be: 1 - Item is open. In case of failure the value will be: 0 -PIN_RESULT_FAIL .
